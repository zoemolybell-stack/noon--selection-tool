# CODEX 项目开发协作指南

Last Updated: 2026-04-13  
Status: active

## Purpose

这份文档定义：
- 用户应该如何给 Codex 提任务
- Codex 在项目开发中应该如何推进
- 什么情况下 Codex 应该持续推进整个板块
- 什么情况下 Codex 应该停下来确认

目标是把协作方式从：

`一个小点 -> 停下等继续 -> 再补一个点`

升级为：

`业务目标 -> 全局拆解 -> 整块闭环 -> 验收`

## 核心原则

默认协作单位不是“一个补丁”，而是“一个完整板块”。

默认模式：
- 用户提供业务目标和约束
- Codex 负责全局拆解、排序、实现、验证、修复、文档同步
- 除非遇到真实决策闸门，否则不应在完成一个小环节后停下来等待“继续”

默认目标：
- 先完成一个可验收的完整板块
- 再让用户做产品判断

不要把用户变成：
- 逐步指挥每一个小修改的人
- 补丁验收员
- 临时项目经理

## 用户任务输入模板

以后每次提需求，优先按这 5 类信息表达。

### 1. 目标

这块最终要解决什么业务问题。

例如：
- 我想做一个可长期使用的补货工作台
- 我想把关键词研究页改成运营真正能用的页面
- 我想让类目爬虫长期稳定运行并且可监控

### 2. 范围

这次要做哪一块，不做哪一块。

例如：
- 这次只做商品机会页，不动首页和关键词页
- 这次只做本地 beta，不做 NAS 发布
- 这次只做类目爬虫链路，不碰 ERP 采购模块

### 3. 优先级

这次最重要的是哪一项：
- 速度
- 稳定性
- 体验
- 长期架构

如果没有特别说明，Codex 默认按：

`业务闭环 > 稳定性 > 结构正确 > 体验润色`

### 4. 约束

明确哪些不能动、必须兼容什么、部署到哪里。

例如：
- 不改 schema
- 不碰 NAS stable
- 必须兼容现有 warehouse
- 不改 main-path Web 文件

### 5. 验收标准

你最终看到什么，才算完成。

例如：
- 打开页面后，运营能在一个页面里完成筛选、比对和详情查看
- 爬虫运行后，Web 在同一轮内能看到新数据
- 用户保存的筛选器能跟着账号跨设备恢复

## 推荐提任务方式

建议直接按下面这个模板发任务：

```text
我要做 [板块名]。

目标：
[这块最终要解决什么业务问题]

范围：
[这次包含什么 / 不包含什么]

优先级：
[速度 / 稳定性 / 体验 / 长期架构]

约束：
[不能动什么 / 必须兼容什么 / 部署到哪里]

验收标准：
[你最终看到什么才算完成]

除非涉及生产部署、数据库 schema 变更、destructive 操作、账号/密钥、或明显路线分叉，否则不要在一个小环节结束后停下，直接推进到整个板块可验收再回来。
```

## Codex 默认执行方式

在任务信息足够后，Codex 默认应这样工作：

1. 理解当前现状
2. 判断当前板块的完整边界
3. 做全局拆解和优先级排序
4. 连续推进实现
5. 运行验证
6. 修复回归
7. 更新文档
8. 直到整个板块达到“可验收状态”后再回报

这意味着：
- 不要做完一个按钮就停
- 不要改完一个样式就停
- 不要修完一个小 bug 就等用户说“继续”

除非任务本身就是极小粒度，否则默认按完整板块推进。

## 板块闭环规则

对 ERP、爬虫、工作台类项目，Codex 应以“板块闭环”为最小交付单位。

例如：

- `商品机会页重构`
  - 不是只改抽屉
  - 而是筛选、表格、详情、说明文案、窄屏策略、回归一起做完

- `类目爬虫稳定化`
  - 不是只修一个文件名 bug
  - 而是运行锁、原子写、resume、sync、验证一并收口

- `ERP 补货一期`
  - 不是只做库存表
  - 而是采购、库存、销量、补货建议的最小闭环

## Codex 应该停下来的情况

只有以下情况，Codex 默认应停下来确认：

- 生产部署
- 回滚
- destructive 操作
- 数据库 schema 变更
- 需要账号、密钥、登录、付费资源
- 有两条以上代价明显不同的实现路线
- 需要用户做产品决策，而不是技术执行

除此之外，Codex 默认应继续推进，而不是等待一句“继续”。

## 代码与验证层面的默认责任

Codex 不仅负责“写代码”，还默认负责：
- 运行必要验证
- 修复自己引入的问题
- 处理直接相关的回归
- 更新 handoff / collab log / 相关说明文档

所以在正常任务中，用户不需要额外逐步要求：
- “继续修复”
- “继续验证”
- “继续补文档”

这些都属于默认闭环的一部分。

## 用户在验收时应该重点看什么

用户更适合判断：
- 业务目标有没有达到
- 页面是否符合运营工作流
- 数据口径是否正确
- 输出结果是否真的有业务价值

用户不需要承担：
- 逐步管理每个技术子任务
- 指挥每一个微小改动
- 替 Codex 决定是否继续做下一个直接相关步骤

## 对当前项目的默认协作要求

对当前 Noon 项目，默认协作模式固定为：

`业务目标明确 -> Codex 全局拆解 -> 完整板块闭环 -> 用户验收`

具体落地要求：
- 爬虫任务：默认推进到链路稳定、入库、同步、验证完成
- Web 任务：默认推进到页面结构、数据契约、浏览器验证、文档同步完成
- ERP 任务：默认推进到最小可运营闭环，而不是单个表单或单个按钮

## 一句话总结

以后最推荐的协作方式是：

`你定义目标、范围、优先级、约束、验收；Codex 负责全局执行与整块闭环；只在真正的决策闸门停下来找你。`
