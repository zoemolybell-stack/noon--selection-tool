一眼看懂：这个项目到底是什么

如果用最简单的话来讲，这个项目就是一套“把 Noon 平台公开商品情报变成经营决策系统”的工具链。

1. 爬虫采集

类目爬虫、关键词爬虫，从前台公开信息持续采集商品、价格、评论、销量信号、配送标签等。
→

2. 阶段库沉淀

先分别写入类目 stage 和关键词 stage，保留原始观测历史。
→

3. 统一仓库

把两边数据整理成一个统一的 warehouse，供 Web 和后续经营逻辑使用。
→

4. Web 工作台

在 ERP 风格的 Web 里做筛选、比对、详情、运行监控和运营判断。
→

5. 后续 ERP / 补货

未来再把商品库、销售、库存、采购、补货计划接进真正的经营中台。

你可以把它理解成：目前已经不是“一个爬虫脚本”，而是一套“数据采集 + 数据仓库 + 运营工作台”的雏形。

数据库：现在是什么结构，为什么先这样做

你现在实际在用哪些数据库

  • data/product_store.db：类目爬虫 stage 库
  • runtime_data/keyword/product_store.db：关键词爬虫 stage 库
  • data/analytics/warehouse.db：统一仓库，Web 主要读取这里
  • data/ops/ops.db：任务、worker、计划、控制状态

换句话说，现在不是“一库打天下”，而是按职责拆开的。

本地和 NAS 不完全一样

  • 本地开发：以 SQLite 为主
  • NAS stable：文档里已经明确切到 retained-data Postgres 作为稳定环境主目标
  • 这不是混乱，而是“开发模式”和“稳定环境模式”不同

为什么一开始用 SQLite 是对的

  • 零运维，复制一个文件就能跑
  • 本地单机开发很快，适合边做边改
  • 早期 schema 经常变，用 SQLite 改起来成本低
  • 爬虫和本地 beta 一开始就是单机优先，不需要先背 Postgres 运维成本
  • 文件级备份、临时测试、分支实验都非常方便

为什么后来又需要 Postgres

  • NAS stable 已经不是“你自己电脑上的脚本”了，而是多服务长期运行
  • 多 worker、多进程、多容器下，Postgres 并发和一致性更稳
  • 适合做正式发布、持久化和团队长期运营
  • 更适合后续 ERP、审批、补货、采购这些真正业务系统的扩展
方案 适合什么阶段 优点 缺点 是否适合你当前项目
纯 SQLite 单机开发、试错期 简单、快、便宜、无运维 多服务并发和正式环境扩展性有限 适合本地 beta
纯 Postgres 稳定环境、多人协作、长期运营 并发稳、正式、易扩展 开发门槛和运维成本更高 适合 NAS stable 和未来 ERP
SQLite + Postgres 双模式 从工具走向平台的过渡期 两边兼顾，节奏合理 理解和维护成本更高 这正是你们现在的现实最优解

目前 Web 是什么架构，为什么这样做

当前 Web 架构

  • 后端：FastAPI
  • 前端：静态 index.html + 多个 JS 模块 + CSS
  • 主要文件：
    • web_beta/app.py
    • web_beta/static/app.js
    • app.home.js、app.keyword.js、app.drawer.js 等

为什么一开始不用 React / Vue

  • 因为项目起点是“快速验证业务链路”，不是先搭一个标准前端工程
  • 静态 HTML/JS 对爬虫项目非常适合：轻、快、部署简单
  • 一开始数据结构和页面信息都不稳定，重前端框架反而会拖慢速度
所以：现在的 Web 不是“落后”，而是典型的 业务先行、工程后补。 这在早期是正确的；只不过到了现在，前端代码体量已经开始逼近“该继续模块化治理”的节点。

当前好处

  • 开发快
  • 部署简单
  • 跟后端和数据层耦合明确
  • 适合快速打样和频繁调整

当前问题

  • app.js 体量偏大
  • 页面逻辑越来越多，维护门槛在上升
  • 长期来看，不利于多人同时改前端

更合理的目标

  • 继续保留 FastAPI + 静态前端大方向
  • 前端按页面和功能拆模块
  • 不必立刻整体重构成 React
  • 先把信息架构和 API 稳定下来

每个核心模块，具体怎么理解

模块 现在是什么 为什么这样设计 当前好坏判断
类目爬虫 深度扫描类目树，抓子类目商品，再逐步写入 stage 类目维度更像“铺网”，适合找大盘机会 成熟度高，是当前最稳定的采集主力
关键词爬虫 围绕关键词 monitor、批次、扩词、质量分层跑 关键词更适合做高意图商品发现和市场热点跟踪 有深度，但比类目链路更复杂，也更需要治理
stage 数据库 保存原始观测和中间结果 防止 Web 直接读 runtime，避免“半成品数据”污染前台 这是正确设计，说明你们不是简单脚本拼接
warehouse 把类目和关键词数据统一成一个可读模型 Web 只读一层，后续 ERP 也只认一层 这是最关键的“平台化步骤”，方向非常对
ops / task center 任务、worker、计划、进度、重试、lease 都在这层 爬虫不再靠手工命令，而是靠运行中心管理 说明项目已开始从工具走向系统
Web 工作台 ERP 风格研究界面，读 warehouse 做筛选、比对、详情 让运营不是看文件，而是看工作台 方向很对，但距离成熟 ERP 还需继续产品化
NAS stable 正式运行环境，有发布、回滚、Cloudflare Access、Postgres 把“本地开发”与“运营环境”隔离 这一步非常关键，说明项目已经能承载真实运行

现在这样做得好的地方

1. 没有把 Web 直接绑死在爬虫结果上

很多项目最大的问题是：爬虫一跑，前台就直接读结果，最后又脏又乱。你们不是这样，已经有 stage 和 warehouse 分层。

2. 已经有“运行中心”意识

任务、worker、调度、sync、health 这些都已经出现了，说明不是临时脚本了。

3. 环境隔离已经成型

local beta 和 nas stable 分开，是长期运行项目必须具备的条件。

4. 文档治理超出一般小项目

白皮书、handoff、collab log 的存在，能大幅减少多窗口 Codex 协作时的失控。

现在还不够好的地方

1. 主入口文件仍然偏大

像 keyword_main.py、web_beta/app.py、web_beta/static/app.js 体量都不小，说明系统正在从“可跑”向“可长期维护”过渡。

2. 根目录工具还没完全并入平台

定价、周报、NIS、分析器很有价值，但现在更像工具箱，不是平台内一个统一流程。

3. 本地与稳定环境心智模型较复杂

因为处在“SQLite 工具时代”向“Postgres 平台时代”的中间地带，所以理解门槛高。

4. 还没进入真正 ERP 阶段

现在强项是“商品 intelligence”，未来还要加采购、库存、销售、补货计划，才能成为完整经营中台。

最终最合理的结构，应该长成什么样

A. 数据接入层

类目爬虫、关键词爬虫、Seller Lab 报表、SFTP、FBPI/订单/库存等适配器。
→

B. 原始与阶段层

保留原始观测、批次、导入记录、失败证据、日志和中间结果。
→

C. 统一经营仓库

以产品、库存、采购、销售、补货为中心的数据模型，供 Web 和 ERP 使用。
→

D. ERP 核心层

采购单、在途、FBN库存、销量、补货建议、审批状态、执行反馈。
→

E. 协同输出层

飞书/企微通知、审批流、日报、预警、任务分派。
层级 你现在有多少 下一步该补什么
数据接入 已经很强 继续增加正式渠道，如 Seller Lab / SFTP / ERP 连接器
统一仓库 已经有明确雏形 把未来采购、库存、销售也并到统一经营模型
运营工作台 已经可用 继续向成熟 ERP 界面演进
经营中台 / ERP 还没完全成型 这会是未来最重要的一步
审批与协同 还没正式接 后续由 ERP 往飞书/企微推送，而不是反过来

如果你只记住 8 句话

  1. 这个项目现在已经不是单纯爬虫，而是“爬虫 + 仓库 + Web 工作台”的系统。
  2. 真正的主系统在 noon-selection-tool/，根目录很多工具是外挂型资产。
  3. 现在的数据库不是乱，而是按职责拆分：stage、warehouse、ops。
  4. 一开始用 SQLite 是对的，因为当时目标是快速落地，而不是先上重型基础设施。
  5. 现在 NAS stable 走 Postgres，也是对的，因为系统已经进入正式运行阶段。
  6. Web 当前是“轻前端 + 强数据层”的路线，这不是落后，而是阶段性选择。
  7. 未来真正的大目标，不是把爬虫继续做大，而是把它接进 ERP，形成采购、库存、销售、补货闭环。
  8. 飞书/企业微信未来更适合做协同和审批，不适合做这个系统的核心数据库。

我对目前项目的整体判断

这是一个已经跨过“玩具项目”阶段、正在从“强工具链”走向“经营平台”的项目。

它现在最强的，不是 UI，也不是某一个脚本，而是：你们已经搭出了一个正确的中间层架构。
真正决定未来上限的，不是再多做一个爬虫，而是把现有的商品 intelligence 系统，继续接到采购、库存、销售和补货计划上。
