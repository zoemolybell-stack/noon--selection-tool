## 以后给 Codex 提任务，优先提供 5 类信息

1. `目标`
- 这块最终要解决什么业务问题

2. `范围`
- 这次做哪一块
- 哪些明确不做

3. `优先级`
- 速度
- 稳定性
- 体验
- 长期架构

4. `约束`
- 不能动什么
- 必须兼容什么
- 部署到哪里

5. `验收标准`
- 最终看到什么才算完成

## 推荐发任务模板

```text
我要做 [板块名]。

目标：
[这块最终要解决什么业务问题]

范围：
[这次包含什么 / 不包含什么]

优先级：
[速度 / 稳定性 / 体验 / 长期架构]

约束：
[不能动什么 / 必须兼容什么 / 部署到哪里]

验收标准：
[你最终看到什么才算完成]

除非涉及生产部署、数据库 schema 变更、destructive 操作、账号/密钥、或明显路线分叉，否则不要在一个小环节结束后停下，直接推进到整个板块可验收再回来。
```

## Codex 默认执行规则

默认不是补丁思维，而是板块闭环思维。

Codex 应默认连续推进：
- 理解现状
- 做全局拆解
- 直接实现
- 跑验证
- 修回归
- 补文档
- 直到整个板块进入可验收状态

不应该：
- 改一个按钮就停
- 修一个小 bug 就停
- 做一个局部样式就等“继续”

## 什么情况下 Codex 应该停下来确认

只有这些情况：
- 生产部署
- 回滚
- destructive 操作
- 数据库 schema 变更
- 需要密钥 / 登录 / 付费资源
- 有多条代价明显不同的路线
- 需要我做明确产品决策

除此之外，默认继续推进。

## 验收时我应该重点看什么

我主要看：
- 业务目标有没有达到
- 页面/workflow 是否顺手
- 数据口径是否正确
- 结果是否真的有业务价值

我不需要：
- 逐步指挥每一个技术子任务
- 替 Codex 做补丁式项目管理

## 一句话原则

`我定义目标、范围、优先级、约束、验收；Codex 负责全局拆解、连续推进、整块闭环；只在真正的决策闸门停下来。`
